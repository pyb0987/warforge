# 카드 설계 접근법

## 새 테마 카드 설계 시 사고 순서

```
1. 질문: "이 테마의 파워는 어디서 오는가?"
   → themes.md 프레임워크에서 핵심 축 확인
   → 한 문장으로 답할 수 있어야 함

2. 유닛 풀 스탯 확인
   → units-{theme}.md에서 평균 ATK/HP, 태그 분포 확인
   → 이 수치가 메커니즘 방향을 결정
     물량(낮은 개체 스탯) / 소수정예(높은 개체 스탯) / 균형

3. 기존 테마와의 차별점 명시
   → "이 테마는 ___와 달리 ___이다" (대비, NOT 대응)
   → 대응표(역할 매핑) 작성 금지

4. 역할군을 테마 고유 용어로 명명
   → 스팀펑크 용어(체인, 제조, 개량) 재사용 금지
   → 테마 판타지에서 자연스러운 이름 사용

5. 7원칙 대조 검증 (architecture.md)
   → 보상 가시성, 의사결정, 축 중복 등 체크
```

## 안티패턴

```
✗ 기존 테마 역할 대응표 작성
  → "스팀펑크의 조립소 = 드루이드의 ???" 식의 접근 금지
  → 각 테마는 독립적 역할 구조를 가짐

✗ 다른 게임 장르의 가정 투영
  → RTS의 유닛 영구 사망 → 오토배틀러에 적용 (실패 #1)
  → 반드시 이 게임의 실제 메커니즘 확인 후 설계

✗ 키워드(번식/부화 등)를 역할로 승격
  → 키워드는 이벤트 타입이지 카드 역할이 아님
  → "번식 카드"가 아니라 "이 카드가 하는 일"로 정의
  → 드루이드에서 "번식(물량)"을 핵심으로 제안한 오류 (실패 #2)

✗ 역할 수를 다른 테마와 맞추기
  → 스팀펑크 9역할이라고 드루이드도 9역할일 필요 없음
  → 테마 메커니즘 복잡도에 따라 역할 수 자연 결정
```
