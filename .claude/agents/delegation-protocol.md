# Delegation Protocol — 경량 모델 작업 위임

모든 에이전트가 공통으로 사용하는 작업 위임 규칙.
판단은 자신이, 실행·검증은 경량 서브에이전트에 위임한다.

## 위임 판정 기준

위임 가능 조건 **3가지를 모두 충족**해야 한다:

| 조건 | 설명 | 예시 |
|------|------|------|
| **입력 명확** | 실행할 작업이 모호함 없이 기술 가능 | "이 명령어를 실행하라", "이 두 파일을 대조하라" |
| **판단 불필요** | 결과 해석에 도메인 전문성이 필요 없음 | pass/fail, 수치 추출, 목록 생성 |
| **출력 고정** | 반환 포맷이 사전 정의됨 | JSON, pass/fail + 사유, 불일치 목록 |

하나라도 미충족이면 직접 수행한다.

## 모델 선택

| 모델 | 용도 | 기준 |
|------|------|------|
| **haiku** | 실행 + 수집 + binary 판정 | 위임 3조건 모두 충족 |
| **sonnet** | 요약 + 경미한 판단 + 구조화 | 입력 명확 + 출력 고정이나, 약간의 해석 필요 |
| **opus** | 분석 + 창의적 판단 + 트레이드오프 | 위임하지 않음 (자신이 수행) |

의심스러우면 한 단계 위 모델을 사용한다.

## 위임 프롬프트 구조

서브에이전트 호출 시 반드시 아래 4요소를 포함한다:

```
## Task
{무엇을 하는지 1~2문장}

## Input
{파일 경로, 명령어, 데이터 등 — 구체적으로}

## Success Criteria
{pass/fail 판정 기준 — binary로 판정 가능하게}

## Output Format
{정확한 출력 포맷}
```

### 출력 포맷 템플릿

**명령어 실행형:**
```json
{"status": "pass|fail", "errors": ["에러 메시지"], "summary": "1줄 요약"}
```

**대조 검증형:**
```json
{"status": "pass|fail", "mismatches": [{"field": "...", "doc": "...", "code": "..."}]}
```

**수집형:**
```json
{"items": [...], "count": N}
```

## 묶음 위임 (Batch Delegation)

서브에이전트 1회 호출 비용 > 명령어 1개 실행 비용이므로,
관련 작업은 **하나의 서브에이전트에 묶어서** 위임한다.

- ✅ "4프리셋 모두 실행하고 결과 보고" → haiku 1회
- ❌ "basic 실행" → haiku, "factory 실행" → haiku, ... → haiku 4회

묶음 기준: 같은 도구를 사용하고, 독립적으로 실행 가능한 작업.

## 병렬 위임

독립적인 작업은 **동시에 여러 서브에이전트**를 spawn한다:

```
balancer (opus):
  → haiku 서브 A: sim 4프리셋 실행 + 에러 체크    ← 병렬
  → haiku 서브 B: 설계 문서 수치 추출              ← 병렬
  → (결과 회수 후) 분석 + 판단은 자신이 수행
```

## 위임 불가 작업

아래는 항상 직접 수행한다:
- 밸런스 분석/원인 추론 (도메인 판단)
- 설계 트레이드오프 비교 (창의적 판단)
- 규칙 진화 제안 (메타 판단)
- 사용자 확인이 필요한 변경 (권한 판단)
- 에러 원인 분석 (디버깅)

## 위임 실패 처리

서브에이전트가 실패하거나 예상 밖 결과를 반환하면:
1. 동일 작업을 한 단계 위 모델로 재시도 (haiku → sonnet)
2. 재시도도 실패하면 직접 수행
3. 반복 실패 패턴이면 위임 불가 작업으로 분류
